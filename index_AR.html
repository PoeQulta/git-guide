إنشاء مستودع جديد

قم بإنشاء مجلد جديد، افتحه وقم بتنفيذ الأمر
git init
لإنشاء مستودع git جديد.

تسجيل الخروج من المستودع

لإنشاء نسخة عمل من مستودع محلي، قم بتشغيل الأمر
git clone /path/to/repository
عند استخدام خادم بعيد، سيكون الأمر الخاص بك
git clone username@host:/path/to/repository

مسار العمل

يتكون المستودع المحلي من ثلاث "أشجار" تتم صيانتها بواسطة git. الأولى هي دليل العمل الذي يحتفظ بالملفات الفعلية. الثانية هي المؤشر الذي يعمل كمنطقة تنظيم وأخيرًا HEAD الذي يشير إلى آخر التزام قمت به.

إضافة والالتزام

يمكنك اقتراح التغييرات (إضافتها إلى المؤشر) باستخدام
git add <filename>
git add *
هذه هي الخطوة الأولى في مسار عمل git الأساسي. لالتزام هذه التغييرات فعليًا، استخدم
git commit -m "رسالة الالتزام"
الآن تم الالتزام بالملف إلى HEAD، ولكن ليس في المستودع البعيد الخاص بك بعد.

pushing changes

التغييرات دلوقتي في HEAD بتاع النسخة المحلية بتاعتك. عشان تبعت التغييرات دي للـ remote repository بتاعك، نفّذ
git push origin master
غيّر master لأي branch تاني عايز تبعت عليه التغييرات.

لو مكنتش clone مستودع موجود وعايز توصل الـ repo بتاعك لخادم بعيد، لازم تضيفه باستخدام
git remote add origin <server>
دلوقتي تقدر تدفع التغييرات للخادم البعيد اللي اخترته

branching

الفروع بتستخدم لتطوير الميزات بشكل معزول. فرع master هو الفرع "الافتراضي" لما تعمل مستودع. استخدم فروع تانية للتطوير وادمجها تاني في master لما تخلص.

اعمل فرع جديد اسمه "feature_x" وانتقل ليه باستخدام
git checkout -b feature_x
ارجع لـ master
git checkout master
واحذف الفرع تاني
git branch -d feature_x
الفرع مش متاح للآخرين إلا لما تدفع الفرع للـ remote repository
git push origin <branch>

update & merge

عشان تحدث الـ repo المحلي بتاعك لأحدث commit، نفّذ
git pull
في دليل العمل بتاعك عشان تعمل fetch و merge للتغييرات البعيدة.
عشان تدمج فرع تاني في الفرع النشط بتاعك (مثلاً master)، استخدم
git merge <branch>
في الحالتين git بيحاول يدمج التغييرات تلقائيًا. للأسف، ده مش دايماً بيبقى ممكن وبيؤدي لـ conflicts. انت المسؤول عن دمج الـ conflicts دي يدويًا بتعديل الملفات اللي git بيعرضها. بعد ما تعدل، لازم تأشر عليهم كأنهم مدمجين باستخدام
git add <filename>
قبل دمج التغييرات، ممكن كمان تعاينها باستخدام
git diff <source_branch> <target_branch>

tagging

يُفضل تعمل tags لإصدارات البرمجيات. دي فكرة معروفة، وبتكون موجودة في SVN كمان. تقدر تعمل tag جديد اسمه 1.0.0 بتنفيذ
git tag 1.0.0 1b2e1d63ff
الـ 1b2e1d63ff بتمثل أول 10 أحرف من commit id اللي عايز تشير ليها بالـ tag. تقدر تجيب الـ commit id من خلال النظر في...

log

في أبسط أشكاله، تقدر تدرس تاريخ المستودع باستخدام..
git log
تقدر تضيف كتير من البرامترات عشان تخلي الـ log يظهر بالشكل اللي عايزه. عشان تشوف بس الـ commits لمؤلف معين:
git log --author=bob
عشان تشوف log مضغوط جدًا بحيث كل commit يكون في سطر واحد:
git log --pretty=oneline
أو يمكن عايز تشوف شجرة ASCII لكل الفروع، مزينة بأسماء الـ tags والفروع:
git log --graph --oneline --decorate --all
عشان تشوف بس الملفات اللي اتغيرت:
git log --name-status
دي بس بعض البرامترات اللي ممكن تستخدمها. لمزيد من المعلومات، شوف
git log --help

replace local changes

في حالة عملت حاجة غلط، اللي طبعًا مش بيحصل أبداً ;)، تقدر تستبدل التغييرات المحلية باستخدام الأمر
git checkout -- <filename>
ده بيستبدل التغييرات في شجرة العمل بتاعتك بآخر محتوى في HEAD. التغييرات اللي اتضافت بالفعل للمؤشر، والملفات الجديدة، هتفضل موجودة.

لو بدل كده عايز تلغي كل التغييرات والـ commits المحلية، اعمل fetch لأحدث تاريخ من السيرفر ووجّه فرع master المحلي بتاعك عليه كده
git fetch origin
git reset --hard origin/master

useful hints

الواجهة الرسومية المدمجة لـ git
gitk
استخدم خروج ملون لـ git
git config color.ui true
اعرض الـ log في سطر واحد لكل commit
git config format.pretty oneline
استخدم الإضافة التفاعلية
git add -i
